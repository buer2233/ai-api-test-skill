# 方式4：pytest 报错驱动（维护）

> 触发条件：任务类型为“维护”，且用户选择方式4，或表达“直接跑 pytest / 按 pytest 报错修 / 运行目标用例看报错维护 / 跑到通过”。

## 强制读取要求

进入本方式后，AI 必须先读取：

1. `doc/coding_style_guide.md`
2. `doc/maintenance_prompt_context.md`
3. 本文件 `doc/mode_maintenance_pytest_driven.md`
4. 用户指定的 `[接口用例文件]` 与 `[接口用例位置]` 对应上下文

## 适用场景

- 用户已明确指定待维护的单个或多个用例
- 用户希望 AI 直接执行 pytest，根据真实报错定位并修复
- 没有最新抓包、参考样本或 cURL，但当前失败日志足以驱动维护
- 用户未明确说明用例报错原因，但允许 AI 通过执行目标 pytest 分析真实失败原因

## 执行步骤

1. **锁定目标用例**：只根据 `[接口用例文件]` 和 `[接口用例位置]` 定位待维护用例，不要求用户补 `[接口方法文件]` / `[接口方法位置]` / `[用例名]`。
2. **读取目标上下文**：读取目标用例全文、所属测试类头部、fixture、相关 `self.xxx` 实例化和导入。
3. **组装最小 pytest 命令**：优先用目标函数名或用户给出的 pytest `-k` 关键字执行最小范围 pytest；工作目录、`PYTHONPATH` 与命令模板按 `config.json` 的 pytest 配置处理。

   **⚠️ pytest 配置要求（关键）**：
   - **工作目录**：读取 `paths.pytest_workdir`
   - **PYTHONPATH**：读取 `pytest.pythonpath`
   - **命令模板**：读取 `pytest.command_template`，默认 `pytest {target} -v --tb=short`
   - **标准命令格式**（由配置和当前 shell 渲染；Windows/PowerShell 使用 `Set-Location` + `$env:PYTHONPATH`，Bash 使用 `cd` + `PYTHONPATH=...`）：
     ```powershell
     Set-Location -LiteralPath "<paths.pytest_workdir>"; $env:PYTHONPATH="<pytest.pythonpath>"; pytest <target> -v --tb=short
     ```
     ```bash
     cd "<paths.pytest_workdir>" && PYTHONPATH="<pytest.pythonpath>" pytest <target> -v --tb=short
     ```

4. **按真实报错定位**：根据 traceback、断言差异、接口响应、导入错误、fixture 错误或返回层级错误判断维护点。

   **报错日志取用规则（关键）**：
   - pytest 执行过程中可能夹杂接口请求/响应日志、业务日志或接口内部打印的错误内容。
   - 分析失败原因时，**只参考用例执行中断处的最后一个 pytest 报错日志内容**，包括最后的 traceback、assertion diff、exception、fixture/import error 等。
   - 不参考执行过程中穿插出现的接口报错日志；这些日志只能作为背景信息，不能作为判断 BUG 或维护类型的直接依据。

5. **充分分析并分类处理**：（⚠️这是最重要的一步，必须严格执行并分析出问题分类）执行 pytest 后，必须先给出基于最后一个中断报错的分析，再按以下三类之一处理；不能在依据不足时盲目判断。

   ### 第一类：功能 BUG，需反馈开发处理

   适用条件：
   - 最后一个 pytest 报错能明确证明接口真实行为与产品/接口契约不一致。
   - 请求参数、前置数据、登录态、用例编排和断言依据经核对后没有明显问题。
   - 返回内容表现为服务端异常、错误码不合理、数据缺失/错误、权限/状态流转异常等，并且能解释为什么属于功能问题。

   执行流程：
   1. 停止修改用例，不为了通过 pytest 改写断言或绕过真实问题。
   2. 返回失败接口、关键请求信息、最后一个中断报错、真实返回摘要。
   3. 给出判定为 BUG 的分析原因和结论。
   4. 提醒用户在页面或接口侧复测，并将结果反馈给开发处理。

   ### 第二类：用例待维护

   适用条件：
   - 最后一个 pytest 报错能明确指向返回结构、字段名、字段类型、枚举值、排序、默认值、必填参数、请求参数结构或接口方法返回层级发生变化。
   - 能确认当前失败是自动化用例/接口方法与最新接口行为不一致，而不是产品功能异常。

   执行流程：
   1. 使用 `/test-fixing` 技能流程维护目标用例或被 traceback 明确指向的接口方法。
   2. 优先做最小补丁，只修改目标用例或明确相关的接口方法。
   3. 默认不要在第一次维护时使用 `/Debugging` 技能；只有当 `/test-fixing` 流程无法解决、维护遇到困难，或无法明确失败前后接口调用、请求/响应层级、接口方法返回层级和调用栈关系时，才切换使用 `/Debugging` 技能辅助排查。
   4. 使用 `/Debugging` 技能时，应在最后一个 pytest 报错位置、报错前的关键接口调用位置或疑似返回层级变化位置设置断点；`/Debugging` 执行到断点后，优先获取当前执行堆栈、调用链、关键局部变量、请求 payload、接口响应和方法返回值，再据此判断维护点。
   5. `/Debugging` 获取的信息只用于补充定位，不替代最后一个 pytest 中断报错作为分类依据；确认是用例待维护后，仍按最小补丁原则修改目标用例或 traceback 明确指向的接口方法。
   6. 如果需要抓包确认最新接口请求/响应，则读取并执行 `doc/mode_maintenance_capture_driven.md` 的抓包维护流程。
   7. 每轮修改后重新执行同一最小范围 pytest，直到通过或重新分类为 BUG/不明确。
   8. 最终输出维护点、pytest 命令、关键报错、修复说明和最终结果。

   ### 第三类：不明确是否为 BUG

   适用条件：
   - 最后一个 pytest 报错不足以证明是功能 BUG，也不足以证明是用例待维护。
   - 缺少页面操作上下文、业务规则、最新接口契约、环境数据状态或必要抓包。
   - 失败可能由测试数据、环境状态、权限配置、页面流程变化、接口异步状态等多种原因造成。

   执行流程：
   1. 不修改用例，不强行调整断言或参数。
   2. 基于用例代码、接口信息和最后一个中断报错，整理详细的页面复现步骤。
   3. 返回用户需要在页面自行操作验证的内容，说明需要判断是功能 BUG 还是用例需维护。
   4. 等用户反馈“待维护”或提供最新抓包/页面结果后，再进入对应维护流程。

6. **最小补丁修复**：仅当分类结果为“用例待维护”时执行。优先修复目标用例；只有 traceback 或调用链明确指向接口方法时，才读取并维护对应接口方法。
7. **循环执行 pytest**：每轮修复后重新执行同一最小范围 pytest，直到通过，或根据最后一个中断报错重新分类为功能 BUG / 不明确是否为 BUG / 环境、账号、网络、会话等非代码问题。
8. **记录修复过程**：最终输出必须包含执行命令、关键报错、分析分类、修改点和最终 pytest 结果；若未修改代码，必须明确说明原因和下一步需要用户确认的信息。

## 维护原则

- pytest 报错是维护依据，不把失败用例扩展成新增任务。
- 不因缺少新增任务 5 项信息而打回维护任务。
- 不扩大运行范围；只有目标用例通过后，才按风险决定是否补充目标类或 marker 级别冒烟。
- 如果连续两轮报错指向不同公共方法，先判断公共影响面，再修改共享接口方法。
- 只有能明确分析出是 BUG 的场景才按功能 BUG 返回；依据不足时统一归为“不明确是否为 BUG”。
- 不明确的报错类型必须先返回复现步骤和待确认点，等待用户确认后再维护。
