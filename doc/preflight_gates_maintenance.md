# 维护任务前置门禁

---

## 🚨 前置必跑 0：入口前置（由 hook 自动执行）

skill 被触发瞬间，项目级 `.claude/settings.json` 的 `PreToolUse` hook 自动执行 `hooks/preflight_hook.py`（仅对 `api-test-E10` 生效），调用 `tools/preflight_check.py` 并把结果以 `additionalContext` 注入上下文，AI 直接原文回显即可、无需再手动调用。

完整时序与设计取舍见 `flow_chart/flow.md` 附录 A。

---

## 🚨 前置必填 A：维护任务信息

维护已有接口方法或 pytest 用例前，AI **在做任何调用、分析、扫描、抓包、写代码之前**，只强制校验 `[接口用例文件]` 和 `[接口用例位置]` 两项。

维护任务不强制要求用户提供新增任务的 `[接口方法文件]` / `[接口方法位置]` / `[用例名]`。只有定位后确认必须修改接口方法且无法从现有用例反查时，才追问接口方法范围。

### 必填信息清单与打回模板

维护任务校验发现缺项时，不要强制要求新增任务 5 项，直接照抄以下模板返回：

```markdown
请先提交本次维护任务的必填信息，补齐后我才能开始工作：

# 本次维护任务信息
- `[接口用例文件]` = `填写接口用例所在文件路径`
- `[接口用例位置]` = `填写具体的待维护的单个/多个用例，例如：test_xxx / 某测试类下的多个用例 / 第456行附近的 xxx 用例`
```

### 校验规则

- `[接口用例文件]` 必须能定位到具体 pytest 用例文件。
- `[接口用例位置]` 必须能定位到具体待维护的单个或多个用例，可为函数名、类名 + 用例范围、行号附近、pytest `-k` 关键字，或用户明确列出的多个用例。
- 如果用户要求"AI 直接跑 pytest 后按报错维护"，但未提供 pytest 命令，AI 可按 `[接口用例文件]` 与 `[接口用例位置]` 自行组装最小范围 pytest 命令。
- **以下情况视为"未提供"**，必须打回让用户补齐维护两项：
  - 只给报错日志，但没有说明哪个用例文件。
  - 只给用例文件，但没有说明维护哪个单个或多个用例。
  - `[接口用例位置]` 只有"都改一下"、"相关用例"等无法定位的泛泛表述。
- **以下情况视为"已提供"**，才继续执行后续流程：
  - 两项必填项全部存在，且均为具体真实内容。
  - 用户在同一会话上文里已给过这些信息（可沿用，不需要重复提交）。
  - 用户明确说"按上次的维护范围继续" / "继续维护同一个用例" 等可以明确锚定上文的表述。

### 例外情形

以下场景允许**不走必填校验**（AI 可直接响应，但仍应在回复末尾提醒用户"正式维护时请先提交本次维护任务信息"）：

- 纯查询/分析类：读索引、查已实现方法、看文件、问功能域分类、聊架构。
- 工具/环境类：检查抓包服务器、启动/停止抓包、扫描索引、生成勾选草稿。
- 诊断类：看日志、分析报错、解释现有代码行为。

一旦任务进入"维护接口方法或用例"阶段，维护任务 2 项必填校验立即生效。

## 🚨 前置必填 B：维护方式选择

维护任务信息校验通过后，如果用户**尚未在任务描述中明确指定维护方式**，AI **必须主动提问**让用户在以下四种方式中选择一个。

### 维护任务四选一菜单（AI 照抄提问，不要修改格式和任何内容）

> 请选择本次用例的维护方式（回复数字即可）：
>
> **① 维护方式1：抓包驱动** —— 从最新抓包 JSONL 回溯链路，对照并维护已有用例
>   - 适用：业务链路变化较大 / 多接口联动 / 需要同步最新请求路径
>   - 需要：浏览器配好 mitmproxy 代理，证书已装，或已有 latest.jsonl
>
> **② 维护方式2：参考已有用例** —— 参考仓库内相似用例，对已有用例做局部维护
>   - 适用：同类用例结构稳定 / 只改参数、fixture、断言或调用方式
>   - 需要：用户告知参考用例或同类样本
>
> **③ 维护方式3：cURL 手工** —— 用户手动粘贴 cURL 请求 + 接口返回值，AI 据此维护
>   - 适用：少量接口变更明确 / 不需要完整抓包回溯
>   - 需要：用户提供每个接口的 cURL 命令 + 对应响应体
>
> **④ 维护方式4：pytest 报错驱动** —— AI 直接执行目标用例 pytest，并按真实报错自行维护到通过
>   - 适用：用户只指定待维护用例，希望 AI 根据 pytest 报错定位并修复
>   - 需要：已提供 `[接口用例文件]` 和 `[接口用例位置]`
>   - 依赖：默认优先使用 `/test-fixing`；只有 `/test-fixing` 无法解决、维护遇到困难或调用栈/前后接口信息不明确时，才使用 `/Debugging` 断点调试辅助定位

### 方式选择的默认推断规则（免提问）

用户任务描述里若出现以下**明确信号**，AI 可以**不再提问**，直接按对应方式执行：

| 信号 | 推断方式 |
|---|---|
| "抓包已就绪" / "用抓包" / "我刚操作完" / "读 latest.jsonl" / "同步最新链路" | 方式1 |
| "参照 xxx_api.py 的 test_xxx" / "仿照已有" / "参考现有用例修复" / "复制用例改参数" | 方式2 |
| 消息里已粘贴 `curl ...` 命令 / "这是我的接口和返回" + JSON | 方式3 |
| "直接跑 pytest" / "按 pytest 报错修" / "你运行用例看报错维护" / "跑到通过" | 方式4 |

若信号模糊或冲突，仍按"维护任务四选一菜单"询问用户。

### 方式选定后的记录要求

AI 在 TodoWrite 首项必须显式记录任务类型、方式编号与 2 项必填任务信息，例如：

```text
[维护-方式4] 维护目标用例（用例文件：xxx，目标：test_xxx）
```

## 项目根定位说明（必须先初始化 config）

本 skill 面向通用 `python + pytest + requests` 接口自动化项目。正式维护前，用户必须先在 `config.json` 中配置：

1. `project_root`
2. `paths.api_method_dirs`
3. `paths.test_case_dirs`
4. `paths.pytest_workdir`
5. `paths.runtime_temp_dir`
6. `api_index.scan_dirs`
7. `pytest.pythonpath` / `pytest.command_template`

维护任务信息校验通过后，AI 不从 `[接口用例文件]` 反推项目根；所有工具统一通过 `skill_utils.project_root.resolve_project_root()` 读取 `config.json.project_root`。

如果运行时出现"找不到项目根"或"扫描目录不存在"，按以下顺序排查：

1. `config.json.project_root` 是否已填写并指向真实目录。
2. `api_index.scan_dirs` 是否相对 `project_root` 可解析为真实目录。
3. `paths.pytest_workdir` 是否相对 `project_root` 可解析为真实目录。
