# 维护专用提示词上下文

> 仅在“维护已有接口方法 / 用例”任务中读取。**不要**在新增任务里加载本文件。

## 这份上下文解决什么

- 把“新增”与“维护”彻底分开，避免新增语义污染维护判断。
- 让 AI 在维护场景里优先判断：改用例、改接口方法、补链路、还是做定点修复。
- 给四种维护方式提供统一的判断基线，减少来回询问。

## 维护任务的关键词信号

以下信号默认按“维护”理解：

- 维护、修复、回归、同步最新链路
- 调整参数、调整断言、更新 fixture、更新依赖字段
- 抓包回溯、最新接口链路、最新请求路径
- 只有一两个接口变更、定点修补、快速修复
- 既有用例失效、方法返回结构变化、接口链路改动

如果用户只说“帮我改一下”，但没有给出变更范围，先追问影响接口、影响用例、是否有最新抓包或 cURL。

## 维护任务的优先判断顺序

1. 先找现有实现，不先假设要新建。
2. 先判断是“只改用例”还是“方法 + 用例一起改”。
3. 若变更范围大、链路变动多，优先抓包回溯。
4. 若有明确的参考用例，优先参考用例维护。
5. 若只有一两个接口且变更清晰，优先 cURL 快修。
6. 若用户希望 AI 自行运行目标用例并按报错修复，使用 pytest 报错驱动维护；该方式默认优先走 `/test-fixing`，只有 `/test-fixing` 无法解决、维护遇到困难或前后接口/调用栈信息不明确时，才使用 `/Debugging` 断点调试辅助定位。

## 维护场景专用信息

维护任务开始前，优先收集这些信息：

- 受影响的接口方法文件
- 受影响的用例文件
- 最新链路、最新抓包或 cURL
- pytest 失败日志或允许 AI 直接运行 pytest 的指令
- 变更前后差异点
- 需要保留的旧断言与需要替换的新断言
- 方式4维护困难时，可通过 `/Debugging` 在最后报错位置或关键接口调用处打断点，获取执行堆栈、调用链、局部变量、请求 payload、接口响应和方法返回值

## 维护任务的边界

- 不把新增任务里的“生成新用例”语义带进来。
- 不把参考新增方式的扩展规则硬套到维护任务里。
- 不清楚是否影响新增接口时，先问用户，不直接扩散改动面。
- 不在第一次 pytest 报错维护时直接使用 `/Debugging`；它是 `/test-fixing` 失败或信息不足时的补充定位手段。
