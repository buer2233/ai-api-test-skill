# 方式3：cURL 手工（维护）

> 触发条件：任务类型为“维护”，且用户选择方式3，或消息中已粘贴 `curl ...`，且变更范围较小、接口改动明确。

## 强制读取要求

进入本方式后，AI 必须先读取：

1. `doc/coding_style_guide.md`
2. `doc/maintenance_prompt_context.md`
3. 本文件 `doc/mode_maintenance_curl_manual.md`
4. 用户提供的每条 cURL 与对应真实响应体

## 适用场景

- 明确只有一两个接口改动
- 不需要完整抓包回溯
- 需要快速、定点地修复现有用例或方法

## 执行步骤

1. **收齐 cURL 与响应**：每个待维护接口都要有成对的 cURL + 真实响应体。
2. **解析变更点**：提取 method、URL、headers、body，并识别与旧实现的差异。
3. **对照现有方法**：查索引，判断复用、补丁式修改，还是必要时新增替代方法。
4. **定点修补用例**：只改受影响的用例、断言、参数或方法调用，不扩散到无关场景。
5. **断言以真实响应为准**：维护场景尤其要避免沿用过期断言。
6. **执行 pytest**：优先跑受影响用例。

## 维护原则

- cURL 是修补依据，不是新建任务语义。
- 若只有局部字段变化，优先局部修补，不重写整条链路。
- 如果响应结构变化导致断言失效，先以真实响应为准重写断言。
